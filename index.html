— What I build, and why

Useful first. Friction low. Control with the user.

Most software now treats the user as a recurring billing event. Cipher is the opposite — desktop tools that you install once, own outright, and never have to log in to.

Every Cipher product solves a single well-defined job: write with AI, analyze a transcript, store a password, record a screen, compress a video, save one. No feature creep, no SaaS lock-in, no telemetry phoning home. You download an installer from GitHub, run it, and you're working in under a minute.

I write, design, package, and ship every release myself. The work stays small on purpose so it stays sharp.

Principle 01

Problem first, feature second.

Every product starts with a real pain — mine or a user's — written down in one sentence. Features that don't serve that sentence don't ship.

Principle 02

Local before cloud.

Default to running on your machine, against your files, with no network calls. Online features are opt-in, transparent, and never required.

Principle 03

Ship the boring parts.

Installer, updates, release notes, hashes, signing path — the parts that turn a prototype into software people actually keep.

— What I commit to

Not just code. Shipping discipline.

A product is only as good as the boring parts around it. These four commitments apply to every Cipher release.

01

Reliability over features.

A working install, a stable unlock, and an exit that doesn't lose your file beat any new feature on the roadmap. Bugs that touch user data block all other work.

02

Usability is the design brief.

If a feature needs a paragraph of explanation, the UI is wrong, not the user. Every release ends with a clean-pass through the most common flows.

03

Performance, visible.

Cipher tools open fast, idle quietly, and don't sit at 6% CPU when nothing is happening. Memory and startup time are tracked release to release.

04

Shipping cadence.

Releases are tagged, hashed, and documented. Updates land when they're ready, not when a roadmap deadline says so — but they do land.

— Process

From one sentence to installer.

Step 01

Frame the problem

Write the user's pain in one sentence. If I can't, the product isn't ready to start.

Step 02

Prototype and test

Ship the simplest working version. Use it daily for a week before adding anything.

Step 03

UX hardening

Spacing, copy, micro-interactions, edge cases. The pass that turns a prototype into a product.

Step 04

Release polish

Package the installer, publish hashes, write release notes, update the site. Then tag.
